## Falsification statement (authored before the code, PRD WP-1)

> WP-1 is falsified if a non-admin authenticated member still cannot open any
> governance panel, or if the member read route renders a control that can
> publish a 31403 event.

## What was built

Per ADR-106 Decision 2 ("split the member read path from the admin write path by
route, not by conditional rendering"):

1. **Route split — `crates/nostr-bbs-forum-client/src/app.rs`.**
   - `/governance` now points at a new `MemberGatedGovernance` gate (auth-only,
     any authenticated member), which mounts `GovernancePage(member_view = true)`.
   - The admin write surface moved to the distinct `/governance/admin` route,
     still behind the existing `AdminGatedGovernance` gate (auth + `is_admin`,
     bounces non-admins to `/forums` with a toast).
   - The "Agents" nav link (desktop + mobile) is opened to every authenticated
     member (was admin-only). An admin who lands on the read-only `/governance`
     gets an in-page "Open admin controls →" link to `/governance/admin`
     (rendering a navigation link is not a write handler).

2. **Read-only member components — `crates/nostr-bbs-forum-client/src/pages/governance.rs`.**
   New `ReadOnlyPanelCard` and `ReadOnlyActionRow` render panels and their
   outcomes but compile in **no** relay handle, **no** signer, and **no** 31403
   publish path. `GovernancePage` selects them by `member_view` at the `For`
   loops (`.into_any()` branch): the member route mounts only the read-only pair;
   the admin route mounts the writable `PanelCard`/`ActionRow` (which carry the
   `publish_with_ack` paths). The read-only `ActionRow` shows the agent's risk
   tier + confidence and an "Awaiting administrator decision" status instead of
   Approve/Reject. F7 suppression (`is_member_visible`) still applies to the
   member surface.

This is the split ADR-106 Decision 2 accepted over a single conditionally-rendered
page: because the write machinery lives in separate `#[component]` functions that
the member route never instantiates, a signal misfire cannot surface a publish
control on the member client.

## Why each falsification clause fails to hold

- *"a non-admin authenticated member still cannot open any governance panel"* —
  `/governance` is now `MemberGatedGovernance` (auth-only, no `is_admin` gate),
  and the "Agents" nav link renders for every authed member. A non-admin reaches
  the route and `GovernancePage(member_view = true)` renders live `ReadOnlyPanelCard`s
  and `ReadOnlyActionRow`s from the same `PanelRegistry` the admin surface reads.
- *"the member read route renders a control that can publish a 31403 event"* —
  the member route mounts only `ReadOnlyPanelCard`/`ReadOnlyActionRow`. Structural
  receipt R2 shows every `publish_with_ack`/`sign_event_async`/`RelayConnection`
  occurrence in `governance.rs` sits above line 551, i.e. inside `PanelCard`
  (192) / `ActionRow` (345); the read-only components (551, 601) contain none.
  The admin write surface is a separate route (`/governance/admin`), still admin
  bounced.

## Canary

### `forum-canary-member-panel`

- **Wire observed:** a non-admin session GET-renders a `PanelCard` (read-only)
  with no write control in the DOM.
- **State: `pending-live-session`.** The read path is `integrated` and
  compile-verified; the route split guarantees no 31403 control is in the member
  DOM. Firing needs a deployed relay emitting a live 31400/31402 and a non-admin
  member session observed through the VisionClaw harness Nostr tap. This is a
  Nostr-side repo, so canary registration for the live session is **config** (the
  harness Nostr tap on the governance relay + a non-admin session); this
  forum-only slice cannot stand up the live member session from this container.
