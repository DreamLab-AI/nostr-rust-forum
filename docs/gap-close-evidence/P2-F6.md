## What the canon requires (DDD §7a) and how this slice meets it

| Canon clause (DDD §7a) | Implementation |
|---|---|
| §7a.1 A published 31403 may be superseded only by the **original signer** or a **strictly higher governance role**; equal/lower role of a *different* signer MUST NOT supersede | `governance::broker::supersede_authority(original_signer, superseder, original_rank, superseder_rank)` → `OriginalSigner` / `HigherRole` / `Unauthorised`; role ranks via `governance_role_rank` (owner>admin>moderator>reviewer>member). The relay combines whitelist-admin status with `broker_roles` grants (`NostrRelayDO::governance_rank`). |
| §7a.2 Supersession is a **new signed kind-31403** that (1) references the superseded event by `e`-tag, (2) carries the superseder's own `DecisionOutcome`, (3) carries a **stated reason** | A superseding 31403 carries `["e", <superseded_event_id>, "", "supersedes"]`; `extract_supersedes_target` detects it. `BrokerCase::supersede` appends the superseder's own outcome + reason and rejects an empty reason (`MissingSupersessionReason`). **Revoke** = a `Reject` referencing a prior `Approve` (tested). |
| §7a.3 Lifecycle gains **Superseded** (superseding 31403) and **Reopened** (appeal 31402); the effective decision is the most recent authorised 31403 in the chain; superseded events are retained as history | `CaseState::{Superseded, Reopened}` added (as_str/parse/is_terminal). `supersede` → `Superseded`; `reopen` (appeal) → `Reopened`; `Reopened` accepts a fresh `record_decision`. The prior decision is **retained** in `history`, never mutated. |
| §7a Invariant 5 The original event is **never mutated** | The superseded decision is marked via a **projection-derived** `broker_decisions.superseded_by` column; the Nostr event on the relay is untouched. |
| §7a.4 Reaches `integrated` when the forum cites §7a and demonstrates an authorised supersede **accepted** and an unauthorised supersede **rejected by `RelayGovernanceGate`** | The ingest gate `supersession_authorised` rejects an unauthorised superseding 31403 with `blocked: unauthorised supersession` before save/projection; the pure `plan_supersession` seam is unit-tested for accept (original signer / higher role) and reject (equal / lower role). |

**Reconciliation note (honest).** The pre-canon PRD (WP-7) and ADR-106 D6 guessed
the shape as `DecisionOutcome::Supersede { prior_decision_id }` and a move to
`Reopened`. The canon that landed this wave (which ADR-106 D6 explicitly *defers
the lifecycle semantics to*) instead models supersession as a normal
`DecisionOutcome` (e.g. a `Reject` = Revoke) carried on a new 31403 that
references the prior event, moving the case to **Superseded**, and assigns
**Reopened** to the *appeal* path. This slice follows the canon (ADR-002: canon
owns cross-repo semantics; the forum is conformist), so there is no
`DecisionOutcome::Supersede` variant — that would contradict §7a.2. Both
`Superseded` and `Reopened` `CaseState` variants exist, discharging the WP-7
falsification target.

## Member and admin surfaces render the supersession history

`pages/governance.rs` gains a `SupersessionHistory` component mounted on **all
four** route-split surfaces (ADR-106 Decision 2): the admin `PanelCard` /
`ActionRow` and the member read-only `ReadOnlyPanelCard` / `ReadOnlyActionRow`. It
is a pure read of the `PanelRegistry` decision chain (`resolve_decision_chain`) —
it mounts no publish path, so the member route stays write-free (DDD Invariant 1).
Each observed 31403 is listed oldest-first; a superseded decision is dimmed +
struck-through and marked **Superseded**, the effective decision is marked
**Current**, and a superseding row is flagged (`↳`). The store ingests 31403
`ActionResponse` events (previously ignored) and builds the chain keyed by case
`d`-tag; `broker_decisions.superseded_by` is exposed on the decisions read API
(`governance_api.rs`) so a server-fed consumer renders the same chain.
