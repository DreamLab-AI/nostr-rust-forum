## Falsification statement (authored before the code, PRD WP-5)

> WP-5 is falsified if roster administration still requires an out-of-band tool,
> if the client issues no call to any of the nine endpoints, or if a roster
> mutation is not reflected in the tab after it returns.

## What was built

An **Agents** tab in the admin surface (`/admin`), mirroring the
`admin/section_requests.rs` idiom (context captured at component level, passed
into async action closures), calling the auth-worker governance roster endpoints
via NIP-98-signed fetch.

1. **`crates/nostr-bbs-forum-client/src/admin/agents_roster.rs`** (new) ‚Äî
   `AgentsRoster` + `RosterRow`.
   - **List** ‚Äî `GET /api/governance/agents` (`fetch_with_nip98_get_signer`),
     parsed by the pure `parse_agents` helper (active-first, then by name).
   - **Register** ‚Äî `POST /api/governance/agents/register` with
     `{ pubkey, name, description, rate_limit_per_min }`. Because the endpoint is
     `INSERT OR REPLACE ‚Ä¶ active = 1`, the same form doubles as the **rate-limit
     edit** (inline per-row input + Save) and the **reactivate** path.
   - **Deactivate** ‚Äî `POST /api/governance/agents/revoke` with `{ pubkey }`
     (`active = 0`).
   - **Activate** ‚Äî re-register with the row's existing fields (sets `active = 1`).
   - Every mutation re-`GET`s the roster and updates the table in the same
     `spawn_local` (`mutate_then_reload`), so the list reflects the change after
     the call returns without a manual refresh of a different tool.
   - The **authorising principal** (`registered_by`) is derived server-side from
     the NIP-98 signer (the admin), so the register form never sends it; the form
     surfaces it read-only ("recorded as `registered_by` from your signature"),
     and each roster row displays its agent's `registered_by` resolved through the
     profile cache (ADR-106 Decision 3 ‚Äî the principal is the registry's, not a
     self-claim).

2. **Wiring** ‚Äî `AdminTab::Agents` added to
   `crates/nostr-bbs-forum-client/src/admin/mod.rs`; the `Agents` `TabButton` and
   the `AgentsRoster` match arm added to
   `crates/nostr-bbs-forum-client/src/pages/admin.rs`, beside `Sections`. The tab
   is inside the admin-gated `/admin` route (a sibling admin route, per ADR-106
   WP-5), so a non-admin never sees it; the endpoints additionally reject unsigned
   or non-admin calls server-side (`require_admin`).

Endpoints reached (of the nine): `GET agents`, `POST agents/register`,
`POST agents/revoke` ‚Äî the roster-management subset the task names (list,
register with pubkey/name/description/principal, edit rate limit,
activate/deactivate). The remaining six (`provision`, `cases`, `cases/:id`,
`roles/grant`, `roles/revoke`, `roles`) are the case/role surface, not the roster.

## Why each falsification clause fails to hold

- *"roster administration still requires an out-of-band tool"* ‚Äî the Agents tab
  registers, edits the rate limit, and activates/deactivates an agent entirely
  in-app; no external tool is needed.
- *"the client issues no call to any of the nine endpoints"* ‚Äî `AgentsRoster`
  calls `GET /api/governance/agents`, `POST /api/governance/agents/register` and
  `POST /api/governance/agents/revoke` via the NIP-98 signer helpers against
  `auth_api_base()`. (A prior search for `api/governance` in the client returned
  zero hits; it now returns these.)
- *"a roster mutation is not reflected in the tab after it returns"* ‚Äî every
  mutation runs `mutate_then_reload`, which awaits the POST then re-`GET`s the
  roster and sets the signal, so the table shows the change on return. The
  `parse_agents` ordering test proves the reload's decode + ordering contract.

- **Wire observed:** the Agents tab issues a NIP-98-signed call to one of the
  nine endpoints and the roster list reflects the mutation.
- **State: `pending-live-session`.** The tab + NIP-98 calls are `integrated` and
